# Labeling Studio 사용 가이드

이 문서는 OpenVisionLab Labeling Studio를 처음 켰을 때 실제로 어떤 순서로 작업하면 되는지 정리한 가이드입니다.

제가 이 프로그램에서 계속 중요하게 보는 부분은 하나입니다.
라벨을 그리고 끝나는 프로그램이 아니라, **데이터셋을 만들고, 라벨을 저장하고, 학습하고, 그 모델로 다시 검사해보는 흐름**이 한 번에 이어져야 합니다.

객체탐지 작업을 기준으로 설명합니다. 세그멘테이션과 이상탐지는 뒤쪽에 따로 적었습니다.

HTML로 크게 보려면 [labeling-workbench-tutorial.html](labeling-workbench-tutorial.html)을 엽니다.
파일 하나만 다른 PC에 복사해서 보려면 이미지가 포함된 [labeling-workbench-tutorial-standalone.html](labeling-workbench-tutorial-standalone.html)을 사용합니다.

캡처 안의 빨간 번호와 화살표를 먼저 보고, 아래 설명에서 같은 번호의 항목을 확인하면 됩니다.

## 먼저 헷갈리기 쉬운 것

처음 사용할 때 가장 많이 헷갈리는 부분입니다.

| 구분 | 설명 |
| --- | --- |
| 이미지 폴더 | 원본 이미지가 있는 곳입니다. 라벨 파일이 저장되는 곳은 아닙니다. |
| 저장 폴더 | 라벨, 클래스 정보, `data.yaml`, 학습 분할 파일이 저장되는 곳입니다. |
| 저장 라벨 | 사람이 만들었거나 확정해서 저장한 실제 정답 데이터입니다. |
| AI 후보 | 모델이 제안한 결과입니다. 확정하고 저장하기 전까지 정답 라벨이 아닙니다. |
| 학습 모델 후보 | 학습은 끝났지만 아직 검사 모델로 선택하지 않은 모델입니다. |
| 현재 검사 모델 | 지금 `현재 검사` 버튼이 사용하는 모델입니다. |

새 데이터셋인데 이전 라벨이 보인다면 대부분 저장 폴더가 같습니다.
이미지 폴더만 바꾸면 새 실험이 되지 않습니다. 새 실험을 하려면 저장 폴더도 새로 잡아야 합니다.

## 전체 화면

처음 화면은 크게 네 영역으로 보면 됩니다.

![전체 작업 화면](images/annotated/01-overview-1920-annotated.png)

| 위치 | 역할 |
| --- | --- |
| 상단 | 현재 단계, 데이터셋, 저장 폴더, 이미지 폴더, 현재 검사 모델 |
| 왼쪽 | 이미지 큐와 이미지별 작업 상태 |
| 가운데 | 라벨과 AI 후보를 보는 캔버스 |
| 오른쪽 | 현재 단계에서 필요한 작업 패널 |

화면이 복잡해 보일 때는 상단 작업 흐름만 보면 됩니다.

```text
1 데이터셋 -> 2 라벨링 -> 3 추론 검토 -> 4 학습/모델
```

## 1. 데이터셋 만들기

처음에는 데이터셋부터 정합니다.

![데이터셋 확인 화면](images/annotated/02-dataset-wizard-actual-annotated.png)

작업 순서:

1. 상단에서 `1 데이터셋`을 선택합니다.
2. 새 데이터셋을 만들거나 기존 데이터셋을 엽니다.
3. 목적을 선택합니다.
4. 원본 이미지 폴더를 지정합니다.
5. 저장 폴더를 확인합니다.

목적 선택 기준:

| 목적 | 언제 선택하는지 |
| --- | --- |
| 객체탐지 | 대상 위치를 박스로 잡아야 할 때 |
| 세그멘테이션 | 대상의 모양을 mask나 polygon으로 더 정확하게 따야 할 때 |
| 이상탐지 | 이미지 단위로 정상/불량을 먼저 나누는 작업일 때 |

저장 폴더는 꼭 확인해야 합니다.
`Test`, `Test2`, `LineA_202607`처럼 데이터셋 이름은 달라도 저장 폴더가 같으면 이전 라벨을 그대로 읽습니다.

## 2. 클래스 등록

라벨을 그리기 전에 클래스를 먼저 정합니다.

예:

```text
OK
NG
Defect
Scratch
Missing
Contamination
```

객체탐지에서는 클래스 순서가 YOLO txt의 class index가 됩니다.
학습을 시작한 뒤에는 클래스 순서를 함부로 바꾸지 않는 것이 좋습니다.

작업 기준:

- 실제 검사 기준과 같은 이름을 씁니다.
- 정상/불량 기준이 섞이지 않게 정리합니다.
- 클래스가 바뀌면 기존 라벨 파일과 맞는지 확인합니다.
- 저장 폴더가 오늘 작업 데이터셋과 맞는지 확인합니다.

## 3. 이미지 큐에서 작업 이미지 고르기

왼쪽 이미지 큐는 파일 목록이면서 작업 현황판입니다.

![이미지 큐와 상태](images/annotated/01-overview-1920-annotated.png)

자주 보는 필터:

| 필터 | 사용 시점 |
| --- | --- |
| 전체 | 작업 전체를 볼 때 |
| AI후보 | 추론 후 후보가 있는 이미지만 볼 때 |
| 실패 | 저장이나 검사 실패 이미지를 다시 확인할 때 |
| 저장됨 | 라벨 저장이 끝난 이미지를 볼 때 |
| 객체없음 | 대상이 없는 이미지로 처리한 항목을 볼 때 |

`다음 미완료`는 아직 저장하거나 검토해야 하는 이미지로 이동합니다.
현재 이미지에 `저장 필요`가 보이면 아직 파일에 반영되지 않은 상태입니다.

## 4. 박스 라벨 그리기

객체탐지에서 가장 기본이 되는 작업입니다.

![라벨링 워크벤치](images/annotated/03-labeling-workbench-1920-annotated.png)

작업 순서:

1. 상단에서 `2 라벨링`을 선택합니다.
2. 툴바에서 `박스`를 선택합니다.
3. 캔버스에서 대상 영역을 드래그합니다.
4. 오른쪽 저장 라벨 패널에서 클래스와 크기를 확인합니다.
5. 클래스가 틀리면 바꾸고 `적용`을 누릅니다.
6. `라벨 저장`을 누릅니다.
7. 왼쪽 이미지 큐가 `저장됨`으로 바뀌었는지 확인합니다.

라벨 기준은 작업 중간에 계속 바꾸면 안 됩니다.
대상 전체가 들어가되 배경이 너무 많이 들어가지 않게 잡고, 모든 이미지에 같은 기준을 적용합니다.

## 5. 저장 라벨과 AI 후보 구분하기

이 프로그램에서 가장 중요하게 나눈 부분입니다.

![저장 라벨 패널](images/annotated/04-saved-label-panel-1920-annotated.png)

저장 라벨은 학습 데이터입니다.
AI 후보는 모델이 제안한 결과입니다.

둘을 한 화면에서 같이 볼 수는 있지만, 같은 의미가 아닙니다.

![라벨과 AI 후보 비교](images/annotated/05-label-vs-ai-1920-annotated.png)

캔버스 보기 모드:

| 보기 | 의미 |
| --- | --- |
| 라벨 | 저장 또는 편집 중인 라벨만 표시 |
| 추론 | AI 후보만 표시 |
| 모두 | 저장 라벨과 AI 후보를 같이 표시 |

작업 기준:

- AI 후보가 맞으면 확정합니다.
- AI 후보가 틀리면 스킵합니다.
- 확정한 후보는 저장 라벨로 넘어갑니다.
- 마지막에는 `라벨 저장`을 눌러 파일에 반영합니다.

## 6. 저장 라벨을 수정한 뒤 다시 저장하기

이미 저장한 라벨도 수정할 수 있습니다.
하지만 수정했다고 바로 파일이 바뀌는 것은 아닙니다.

수정 직후:

![저장 필요 상태](images/annotated/07-save-required-1920-annotated.png)

이 상태에서는 상단, 왼쪽 이미지 큐, 오른쪽 저장 라벨 패널에 `저장 필요`가 보여야 합니다.

저장 후:

![저장 완료 상태](images/annotated/08-save-done-1920-annotated.png)

저장 후에는 `라벨 저장됨`, `저장됨`, `파일 저장됨` 같은 상태로 돌아옵니다.
이 상태가 확인되면 다음 이미지로 넘어가도 됩니다.

## 7. 템플릿 반복 라벨링

비슷한 제품이 같은 위치에 반복해서 나오면 템플릿 기능을 쓸 수 있습니다.

![템플릿 반복 라벨링](images/annotated/06-template-batch-1920-annotated.png)

이 기능은 모델 학습이 아닙니다.
기준 라벨과 비슷한 모양을 찾아서 후보를 만들어주는 보조 도구입니다.

사용 순서:

1. 기준 이미지에서 박스를 하나 그립니다.
2. 기준 라벨을 저장합니다.
3. `현재 이미지 후보`로 한 장을 먼저 확인합니다.
4. 위치가 맞으면 저장합니다.
5. 조건이 괜찮으면 `전체 이미지 후보`로 이미지 큐를 한 번씩 돕니다.
6. 만들어진 후보를 검토하고 저장합니다.

잘 안 잡히는 경우:

- 기준 박스가 너무 넓습니다.
- 배경이 많이 포함됐습니다.
- 이미지마다 밝기나 크기가 많이 다릅니다.
- 이미 같은 위치에 저장 라벨이 있어서 후보에서 제외됐습니다.

이럴 때는 기준 박스를 더 정확하게 다시 잡고, 전체 이미지 후보를 돌리기 전에 한 장만 먼저 확인합니다.

## 8. 학습 전 데이터셋 점검

학습 버튼을 누르기 전에 데이터셋 점검을 봅니다.

확인할 것:

- 이미지 수
- 라벨 수
- 클래스 수
- train/valid/test 분할
- 클래스별 라벨 수
- 라벨 파일 형식 오류
- 클래스 불균형
- test split 부족 여부

경고가 있어도 학습이 시작될 수는 있습니다.
하지만 데이터가 부족하면 모델 결과를 믿기 어렵습니다. 특히 불량 이미지가 너무 적거나 test가 없으면 모델 비교가 의미 없어질 수 있습니다.

## 9. 학습 시작

학습은 `4 학습/모델 센터`에서 진행합니다.

![학습/모델 센터](images/annotated/09-model-center-1920-annotated.png)

학습 전 확인:

1. Python 실행 파일 경로
2. YOLOv5 프로젝트 경로
3. client script 경로
4. 학습 이미지 크기
5. batch, epoch
6. 데이터셋 점검 결과

학습 중 확인:

- epoch가 진행되는지
- worker 응답 대기 상태인지
- 실패했다면 실패 사유가 패널에 보이는지
- 중지할 수 있는 상태인지

학습 완료:

![학습 완료와 모델 후보](images/annotated/10-training-complete-1920-annotated.png)

학습이 끝나면 새 `best.pt`가 모델 후보로 등록됩니다.
여기서 중요한 점은, 학습이 끝났다고 바로 그 모델로 검사하는 것은 아니라는 점입니다.

## 10. 현재 검사 모델 확인

현재 검사 모델은 지금 `현재 검사` 버튼이 사용하는 모델입니다.

![모델 이력](images/annotated/11-model-history-1920-annotated.png)

학습 후에는 아래 순서로 확인합니다.

1. 학습이 완료됐는지 봅니다.
2. 새 모델 후보가 등록됐는지 봅니다.
3. 후보 검증을 실행합니다.
4. 기존 모델과 비교합니다.
5. 더 낫다고 판단되면 검사 모델로 저장합니다.
6. 상단 `추론 상태`에서 모델명이 바뀌었는지 확인합니다.

이 단계를 나눈 이유는 현장에서 새 모델을 무조건 바로 적용하면 안 되기 때문입니다.
특히 불량 검출에서는 recall이 좋아졌는지, 정상 과검출이 늘었는지 같이 봐야 합니다.

## 11. 현재 이미지 추론 검토

추론 검토는 모델 후보를 확인하는 단계입니다.

![추론 검토 패널](images/annotated/12-inference-dock-1920-annotated.png)

작업 순서:

1. `3 추론 검토`로 이동합니다.
2. `현재 검사`를 실행합니다.
3. 후보가 있으면 위치와 클래스를 확인합니다.
4. 맞으면 확정합니다.
5. 틀리면 스킵합니다.
6. 확정된 후보는 저장 라벨이 됩니다.
7. 마지막에 `라벨 저장`을 누릅니다.

후보가 하나도 없을 때는 먼저 현재 검사 모델과 confidence 값을 확인합니다.
라벨 보기 모드와 추론 보기 모드를 헷갈린 경우도 많습니다.

## 12. 모델 적용 판단

여러 모델을 돌릴 수 있으므로, 새 모델이 생겼다고 바로 바꾸지 않습니다.

![후보 모델 판단](images/annotated/13-model-decision-1920-annotated.png)

판단 기준:

- 같은 test split으로 비교했는가?
- 같은 클래스 목록으로 비교했는가?
- precision, recall이 기존보다 나아졌는가?
- 불량을 놓치는 문제가 줄었는가?
- 정상 이미지를 불량으로 잡는 문제가 늘지 않았는가?

![모델 검사와 적용](images/annotated/14-model-inspect-1920-annotated.png)

근거가 있으면 검사 모델로 저장합니다.
근거가 부족하면 기존 모델을 유지하는 편이 안전합니다.

## 13. 세그멘테이션 작업

세그멘테이션은 박스보다 더 정확한 영역 라벨이 필요할 때 사용합니다.

기본 흐름:

1. 데이터셋 목적을 세그멘테이션으로 선택합니다.
2. 클래스를 등록합니다.
3. polygon, brush, eraser로 영역을 만듭니다.
4. 저장 라벨 패널에서 mask 또는 polygon이 저장됐는지 확인합니다.
5. 저장 후 학습 데이터로 사용합니다.

세그멘테이션 라벨은 객체탐지 라벨과 파일 형식이 다를 수 있습니다.
모델마다 요구하는 형식이 다르기 때문에 학습 전에 저장 결과를 확인해야 합니다.

## 14. 이상탐지 작업

이상탐지는 처음부터 박스를 그리는 작업이 아닐 수 있습니다.

처음에는 아래 기준을 먼저 정합니다.

- 이미지 단위 정상/불량만 필요한가?
- 결함 위치 라벨까지 필요한가?
- 정상 이미지와 불량 이미지가 충분히 분리되어 있는가?
- 사용할 모델이 어떤 입력 형식을 요구하는가?

아직 검증 gate가 없는 이상탐지 기능은 완료 기능처럼 표시하지 않는 것이 기준입니다.

## 15. 하루 작업 체크리스트

라벨링 시작 전:

- 데이터셋 이름 확인
- 이미지 폴더 확인
- 저장 폴더 확인
- 클래스 목록 확인
- 현재 검사 모델 확인

라벨링 중:

- 라벨 기준을 일정하게 유지
- 저장 필요 상태가 남아 있지 않은지 확인
- AI 후보와 저장 라벨을 구분
- 애매한 이미지는 기준을 기록

학습 전:

- 데이터셋 점검 실행
- train/valid/test 분할 확인
- 클래스 불균형 확인
- 저장 필요 이미지가 없는지 확인

학습 후:

- `best.pt` 후보 등록 확인
- 후보 검증 실행
- 기존 모델과 비교
- 검사 모델로 저장할지 결정
- 상단 모델 상태 확인

## 자주 묻는 질문

### 새 데이터셋인데 왜 라벨이 남아 있나요?

저장 폴더가 같을 가능성이 큽니다.
이미지 폴더는 원본 이미지 위치이고, 라벨은 저장 폴더를 따라갑니다.

### 학습이 완료됐는지는 어디서 보나요?

`4 학습/모델 센터`에서 완료 상태와 모델 후보를 봅니다.
`best.pt`가 후보로 등록됐는지도 확인합니다.

### 내가 어떤 모델로 검사 중인지는 어디서 보나요?

상단 `추론 상태`, `학습/모델 센터`, 모델 이력을 같이 봅니다.
상단에 보이는 모델이 현재 검사 기준입니다.

### AI 후보가 보이면 라벨링이 된 건가요?

아닙니다.
AI 후보는 모델 제안이고, 확정 후 저장해야 라벨이 됩니다.

### 저장 버튼을 눌렀는데 다음 이미지로 넘어가도 되나요?

왼쪽 이미지 큐와 상단 상태가 `저장됨` 또는 `라벨 저장됨`으로 바뀌면 넘어가도 됩니다.
`저장 필요`가 남아 있으면 아직 확인이 필요합니다.

## 문서 관리 기준

UI가 바뀌면 이 문서도 같이 바꿔야 합니다.

- README나 튜토리얼에 들어가는 화면 이미지는 반드시 현재 EXE 기준의 최신 UI 캡처를 사용합니다.
- UI가 바뀐 뒤에는 예전 캡처를 그대로 두지 않고 다시 캡처한 뒤 번호와 화살표를 새 위치에 맞춰 다시 넣습니다.
- 주요 버튼 위치가 바뀌면 캡처를 갱신합니다.
- 학습/모델 문구가 바뀌면 학습 완료와 검사 모델 적용 설명을 다시 확인합니다.
- 저장 폴더, 저장 라벨, AI 후보 구분은 문서 앞쪽에 유지합니다.
- standalone HTML은 이미지가 모두 포함되어야 하므로 HTML을 수정한 뒤 다시 생성합니다.
